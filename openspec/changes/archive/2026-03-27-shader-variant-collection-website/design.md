## Context

这是一个新项目，用于收集游戏引擎运行时产生的 shader 变体信息。当前没有现有系统，需要从零构建。

约束条件：
- 使用极简技术栈（Node.js + Express + 纯 HTML/JS）
- JSON 文件存储，但需要考虑性能（使用内存 HashSet 去重）
- 单机部署，端口 8880

## Goals / Non-Goals

**Goals:**
- 提供可靠的变体数据收集和去重机制
- 简洁实用的 Web 界面，支持搜索和分页
- 每条数据立即持久化，不丢失
- HTTP API 供客户端提交和查询数据

**Non-Goals:**
- 不做用户认证/权限控制
- 不做数据库存储（使用 JSON 文件）
- 不做多实例/分布式部署
- 不做数据导出功能（可通过 API 获取）

## Decisions

### 1. 存储方案：JSON 文件 + 内存 HashSet

**选择**: 混合方案 - JSON 持久化 + 内存 Set 去重

```
┌─────────────────────────────────────────┐
│  服务器启动                              │
│  ┌─────────┐      ┌─────────────────┐  │
│  │ HashSet │ ←─── │ 加载 data.json  │  │
│  │ (去重)  │      └─────────────────┘  │
│  └────┬────┘                            │
│       │                                 │
│  新数据到达                              │
│       ▼                                 │
│  ┌─────────┐  存在?                     │
│  │ Set.has │──────▶ 拒绝，返回 exists   │
│  └────┬────┘                            │
│       │ 不存在                           │
│       ▼                                 │
│  ┌─────────┐      ┌─────────────────┐  │
│  │ Set.add │      │ 追加 data.json  │  │
│  └─────────┘      └─────────────────┘  │
└─────────────────────────────────────────┘
```

**为什么不直接用 JSON**:
- 每次去重需要遍历整个数组 O(n)
- 数据量大时性能差

**为什么不用 SQLite**:
- 增加依赖复杂度
- JSON 对于预期数据量足够

**权衡**: 内存占用换取查询速度。如果变体数量超过 10 万条，需要考虑迁移到 SQLite。

### 2. 数据持久化：立即写入 + 追加模式

**选择**: 每条数据到达时立即追加写入 JSON 文件

**为什么不批量写入**:
- 用户要求立即持久化
- 避免服务崩溃时数据丢失

**实现方式**:
- 读取整个文件 → 追加新数据 → 写回文件
- 使用 `fs.writeFileSync` 确保同步完成

**权衡**: 写入性能换取数据安全性。高并发场景可能成为瓶颈。

### 3. 前端方案：单 HTML 文件内嵌 CSS/JS

**选择**: `public/index.html` 包含所有前端代码

**为什么不用框架**:
- 功能简单，不需要复杂状态管理
- 减少构建步骤和依赖
- 易于维护和部署

### 4. 搜索实现：前端实时过滤 + 防抖

**选择**: 数据加载到前端后，在浏览器内过滤

```
输入变化 → 防抖 200ms → 过滤数组 → 重新渲染
```

**为什么不在后端过滤**:
- 数据量不大，前端过滤足够
- 减少 API 请求次数
- 实时响应更好

### 5. 网络监听：支持局域网访问

**选择**: 监听 `0.0.0.0:8880`，允许同一 WiFi 下的所有设备访问

```
┌─────────────────────────────────────────────────────────────┐
│                      WiFi 局域网                             │
│                                                             │
│   ┌─────────┐     ┌─────────┐     ┌─────────┐              │
│   │ 服务器   │     │ 手机    │     │ 其他PC  │              │
│   │ 0.0.0.0 │     │         │     │         │              │
│   │  :8880  │◀────┤ 客户端  │◀────┤ 客户端  │              │
│   └─────────┘     └─────────┘     └─────────┘              │
│        │                                                    │
│        │  http://192.168.x.x:8880                          │
│        ▼                                                    │
│   所有设备可通过服务器内网 IP 访问                            │
└─────────────────────────────────────────────────────────────┘
```

**监听地址对比**:
| 地址 | 效果 |
|------|------|
| `127.0.0.1` / `localhost` | 仅本机可访问 |
| `0.0.0.0` | 所有网络接口，局域网设备可访问 |

**注意事项**:
- Windows 防火墙可能阻止入站连接，首次运行需允许
- 客户端需使用服务器的内网 IP（如 192.168.x.x）访问

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| JSON 文件损坏 | 数据丢失 | 写入前可备份（可选实现） |
| 内存占用过大 | 服务崩溃 | 监控数据量，超过阈值告警 |
| 并发写入冲突 | 数据不一致 | 使用同步写入，单线程处理 |
| 大数据量前端卡顿 | 体验差 | 分页限制单次渲染数量 |
