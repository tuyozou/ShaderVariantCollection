## Context

当前系统是一个 Shader 变体收集服务：
- C# 客户端 (`ShaderVariantClient.cs`) 发送变体日志到 Node.js 服务端
- 服务端 (`server.js`) 存储去重后的数据到 `data.json`
- WebUI (`public/index.html`) 展示和搜索日志

现有数据结构：`{ message, time }`，仅基于 message 去重。

## Goals / Non-Goals

**Goals:**
- 客户端发送时携带资源版本
- 服务端聚合同一 message 的所有版本
- WebUI 展示版本标签并支持按版本过滤
- 保持现有去重逻辑（基于 message）

**Non-Goals:**
- 不支持版本的删除或编辑
- 不迁移老数据（直接清空）
- 不限制 versions 数组大小（预期最多 30 个版本）

## Decisions

### D1: 版本存储为数组而非独立记录

**决策**: 同一 message 的多个版本聚合到 `versions[]` 数组，而非为每个 message+version 组合创建独立记录。

**理由**: 
- 符合需求"显示产生这些日志的所有资源版本"
- 减少数据冗余
- 便于 WebUI 展示聚合视图

**替代方案**: message+version 组合作为唯一键 → 会导致大量重复 message，不便于查看问题影响范围

### D2: 版本按语义化排序

**决策**: versions 数组按语义化版本号排序（1.2.0 < 1.10.0 < 2.0.0）

**理由**: 
- 版本号是资源包版本，符合 semver 格式
- 排序后更易识别问题影响的版本范围

**实现**: 使用自定义比较函数，按 major.minor.patch 数值排序

### D3: 客户端配置而非每次传入版本

**决策**: C# 客户端新增 `resourceVersion` 字段，调用 `Submit(message)` 时自动携带

**理由**:
- 资源版本在整个会话中通常不变
- 简化调用方代码
- 可在 Inspector 中配置

### D4: WebUI 版本过滤使用下拉选择器

**决策**: 搜索框旁新增版本下拉选择器，选项从数据中动态提取

**理由**:
- 版本号是固定集合，下拉比输入更友好
- 避免用户记忆/输入版本号
- 可快速切换查看不同版本的问题

### D5: 版本标签折叠显示

**决策**: 默认显示前 10 个版本，超出部分显示"+N 更多"，点击展开

**理由**:
- 预期最多 30 个版本，全部显示会占用过多空间
- 10 个是合理的折叠阈值
- 保持列表紧凑

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 版本排序对非 semver 格式可能失败 | 排序失败时 fallback 到字符串排序 |
| 老数据不兼容 | 用户确认可以清空，不做迁移 |
| 版本下拉选项过多 | 预期 30 个版本可接受，暂不分页 |
